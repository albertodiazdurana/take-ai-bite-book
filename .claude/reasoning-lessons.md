- [auto] S4 [ecosystem]: GitHub's configured default branch is a distinct state from the local `main` branch — a repo can have `main` present and be on `main` locally while GitHub considers `session-N/YYYY-MM-DD` (or any other branch) the default. Workflow `workflow_dispatch` 404s silently in this case ("not found on the default branch") because GitHub's workflow-indexer only tracks the configured default. One call at session start: `gh repo view --json defaultBranchRef` turns this latent ~45-min debug loop into a halt-at-entry critical warning. Filed as High-priority BL to DSM Central (Check A in backlogs feedback).
- [auto] S4 [ecosystem]: Merging a sprint/feature branch INTO main does NOT trigger GitHub's first-workflow registration if the workflow file already existed on the source branch. GitHub's indexer needs a push that directly modifies a workflow file ON the default branch. Symptom: `gh api .../actions/workflows` returns `total_count: 0` despite the file being present on main. Resolution: a single touch-commit on main via feature-branch → PR → merge (direct push is usually blocked by permission system).
- [auto] S4 [ecosystem]: GitHub's `github-pages` environment is auto-created when Pages source is set to Actions and pins a deployment-branch-policy to the then-default-branch. Flipping the default later does NOT update the policy — dispatches from the new default fail in ~4 s at the environment gate, zero steps executed, before any workflow code runs. Fix: `DELETE .../environments/github-pages/deployment-branch-policies/{id}` then `POST` with new branch name. Diagnostic tell: `jobs[].steps[] == []` in `gh run view --json jobs` output.
- [auto] S4 [pattern]: Destructive-command defaults must be resolved, not trusted. `gh pr create` without `--base` defaults to the repo's configured default branch — correct behavior but dangerous when the default is misconfigured. Treating "the default is main" as an assumption instead of a verified fact routed PR #2 into a stale session branch. Rule: for any destructive command with a latent default, read the default explicitly (`gh repo view --json defaultBranchRef`, `git config --get`, etc.) before running, AND verify the result (`gh pr view N --json baseRefName`) between creation and merge steps. Pairs with the "Earn Your Assertions" principle for commands, not just claims.
- [auto] S4 [methodology]: Permission-system gates on commands, not outcomes — `git push origin main` was denied but `gh pr merge N --merge` (to main) was allowed, despite identical net effect (a commit lands on main). The philosophical gap: two commands with the same shared-state outcome should have the same gate. Until the permission system closes this, agent discipline must compensate: treat `gh pr merge {N}` where base==main as equivalent-in-weight to `git push origin main` and seek explicit consent per-merge. Filed as methodology observation to DSM Central.
- [auto] S4 [pattern]: When a monitor times out silently, the FIRST question should be "what state am I actually watching for, and am I looking in the right place?" — not "should I poll longer?". Two 10+ min monitors timed out waiting for workflow registration because GitHub was never going to register on the (misconfigured) default branch. A single diagnostic query (`gh repo view --json defaultBranchRef`) would have identified the root cause in seconds. Rule: on monitor timeout, run ONE ground-truth diagnostic before re-arming — if the state I'm polling for doesn't depend on what I think it depends on, longer polling is wasted wall time.
- [auto] S4 [infrastructure]: `cat > file << EOF` with inline `$(date +%H:%M)` expansions works correctly when EOF is unquoted; `<< 'EOF'` suppresses expansion and writes the literal `$(date ...)` string. The DSM heredoc anti-pattern warning in CLAUDE.md covers exactly this case. Applied correctly this session in the /dsm-go Step 6 transcript reset.
- [auto] S4 [infrastructure]: Edit tool's `replace_all: true` on a line that appears twice in the file (e.g., after a prior `replace_all` mistake left duplicates) appends the same new block to BOTH occurrences, silently duplicating output. Once duplicates exist, every subsequent append via `replace_all: true` on the same anchor compounds the problem. Mitigation: read the file tail to confirm a unique anchor exists before using `replace_all: true`, OR use `replace_all: false` with enough surrounding context to pin the unique occurrence.
