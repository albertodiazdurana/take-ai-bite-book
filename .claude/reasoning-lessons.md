- [auto] S4 [ecosystem]: Merging a sprint/feature branch INTO main does NOT trigger GitHub's first-workflow registration if the workflow file already existed on the source branch. - [auto] S4 [ecosystem]: GitHub's `github-pages` environment is auto-created when Pages source is set to Actions and pins a deployment-branch-policy to the then-default-branch. Flipping the default later does NOT update the policy — dispatches from the new default fail in ~4 s at the environment gate, zero steps executed, before any workflow code runs. - [auto] S4 [pattern]: Destructive-command defaults must be resolved, not trusted. `gh pr create` without `--base` defaults to the repo's configured default branch — correct behavior but dangerous when the default is misconfigured. Treating "the default is main" as an assumption instead of a verified fact routed PR #2 into a stale session branch. Rule: for any destructive command with a latent default, read the default explicitly (`gh repo view --json defaultBranchRef`, `git config --get`, etc.) before running, AND verify the result (`gh pr view N --json baseRefName`) between creation and merge steps. Pairs with the "Earn Your Assertions" principle for commands, not just claims.
- [auto] S4 [methodology]: Permission-system gates on commands, not outcomes — `git push origin main` was denied but `gh pr merge N --merge` (to main) was allowed, despite identical net effect (a commit lands on main). The philosophical gap: two commands with the same shared-state outcome should have the same gate. Until the permission system closes this, agent discipline must compensate: treat `gh pr merge {N}` where base==main as equivalent-in-weight to `git push origin main` and seek explicit consent per-merge. Filed as methodology observation to DSM Central.

- [auto] S4 [infrastructure]: Edit tool's `replace_all: true` on a line that appears twice in the file (e.g., after a prior `replace_all` mistake left duplicates) appends the same new block to BOTH occurrences, silently duplicating output.
