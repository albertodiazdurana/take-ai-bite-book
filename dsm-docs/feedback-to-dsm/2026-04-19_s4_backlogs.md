## Proposal 1: Explicit default-branch verification at repo-init and at every PR-creation

**Priority:** High
**Complements:** BACKLOG-319 (transcript-hooks) — same class of session-start / action-start guardrail.

### Problem

The project's GitHub repo had `session-1/2026-04-17` configured as the **default branch** (set at repo creation, on 2026-04-17). Sprint 1 proceeded for three sessions on the assumption that `main` was the default branch. The mismatch was invisible until Sprint 1 merged to main (Session 4, 11:00 today) and a workflow_dispatch fire failed with HTTP 404 ("workflow not found on the default branch"). Direct verification via `gh repo view --json defaultBranchRef` confirmed `session-1/2026-04-17` was the configured default.

Consequences observed today:
1. **Wasted ~20 minutes** diagnosing what I thought was a GitHub workflow-registration lag (monitors polling `actions/workflows`, retries, elaborate theories about first-workflow indexing).
2. **Wrong-base PR**: I created PR #2 without `--base main`, and `gh pr create` correctly defaulted to the repo's *configured* default (`session-1/2026-04-17`). The PR merged into a stale branch instead of main, producing a cosmetically dirty fast-forward on a dead branch and leaving main unchanged.
3. **False mental model for Sprint 1 closure**: every merge-to-main assumption in the Sprint 1 plan and in the agent's reasoning was wrong. The "merge Sprint 1 → main" step at 11:00 did land commits on main, but main was never the publication target GitHub cared about.

None of DSM's existing guardrails caught this:
- `/dsm-align` does not verify GitHub's configured default branch matches the local `main` branch the agent has been working against.
- `/dsm-go` Step 0 (session branch setup) operates on the *local* branching model (main as base) without cross-checking the remote's configured default.
- `App Development Protocol` confirms-before-action only covers push/merge events, not repo-config drift.

### Evidence

- Project: dsm-jupyter-book
- Remote: github.com/albertodiazdurana/take-ai-bite-book
- `gh repo view --json defaultBranchRef` → `{"name":"session-1/2026-04-17"}` (observed 2026-04-19 11:30)
- Branches on remote: `main`, `session-1/2026-04-17`, `sprint-1/2026-04-17`, `ops/trigger-workflow-registration` — main existed but was not default
- PR #1 (2026-04-17): merged session-1/2026-04-17 → main. Should have updated default, but GitHub does not auto-switch default branch after a merge.
- PR #2 (2026-04-19 11:24): ops/trigger-workflow-registration → *session-1/2026-04-17* (unintended). Head was correct; base was the repo's configured default.
- 10+ minutes of polling `gh api .../actions/workflows` returned `total_count: 0` because GitHub only indexes workflows on the default branch, and the default branch did not contain `scheduled-build.yml` until PR #2 accidentally added it.

### Proposal

Add two checks, one preventive and one detective:

**Check A (preventive, at project init / `/dsm-go` session-start Step 2a):**
If `GIT_AVAILABLE` is true and the repo has a remote `origin` pointing at GitHub, resolve the configured default branch via `gh repo view --json defaultBranchRef --jq .defaultBranchRef.name`. Compare against the local `main` (or the branch the user considers the "main line" per `.claude/CLAUDE.md` conventions). If they differ, halt with a critical warning:

> "GitHub default branch is `{remote-default}`, but DSM expects `main`. Sprint work targeting main will not be indexed by GitHub Actions, Pages, or dependency bots. Fix: Settings → Branches → change default branch to `main`, or update DSM to track `{remote-default}` as the working base. This is a hard gate — no session work until resolved."

**Check B (detective, at `gh pr create` time inside any agent-invoked PR flow):**
Before `gh pr create`, resolve the intended base explicitly. Always pass `--base {resolved-default}`. After creation, verify with `gh pr view {N} --json baseRefName` and assert it matches the intended target. This turns a latent bug (base drift) into an immediate visible failure.

### Relation to existing guardrails

- BL-319 hardens *per-turn transcript append*. This proposal hardens *per-session repo-state verification* — same safety philosophy, different failure domain.
- Complements the post-merge branch recreation rule (DSM_0.2 §20.8): that rule covers branch state after PR merge, but only if the PR merge landed on the intended target. Check B fills the "did the merge go where we thought?" gap.
- Complements `/dsm-align` Step 8c (path validation): that scans CLAUDE.md for stale filesystem paths. The remote-default-branch is the GitHub analog of a stale path reference, just stored in GitHub's config rather than in a file.

### Cost

- Check A: one extra `gh repo view` API call at session start. Cached for the session. Trivial cost.
- Check B: one extra `gh pr view` call after each `gh pr create`. Trivial cost.

### Priority justification

High. The failure mode silently corrupts Sprint artifacts (commits land on main while GitHub considers session-1 canonical), and the diagnostic symptom (HTTP 404 on workflow_dispatch) looks like a GitHub infrastructure issue rather than a config issue. Unrecovered, this class of bug produces compounding drift: each subsequent PR goes to the wrong base, each subsequent workflow fails to register, each subsequent "merged" state is fictitious. The preventive check (A) turns a silent corruption into a halting error at the earliest possible point.
