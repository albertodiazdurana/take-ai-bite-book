## Goal

Deliver an end-to-end sync pipeline that takes a DSM Central release tag
and produces a deployed book on GitHub Pages. First proof via manual
trigger (`workflow_dispatch`); scheduled cron is intentionally deferred
to Sprint 2.

## Inputs (settled)

- **Decision 0001:** Tool = Jupyter Book 2 (MyST Document Engine).
- **Decision 0002:** Sync = scheduled cron (pull-from-book-repo).
- **Decision 0003:** Content scope = flat `*.md` from Central root;
  LICENSE = footer link; tag filter = strict semver; version display =
  title + footer with date.
- **R3 verify-in-implementation items:** MyST build time, largest-page
  load, `myst.yml` footer-template feasibility.

## Exit criteria (Sprint 1 is done when all of these are true)

1. Site deploys successfully at DSM Central **v1.5.2** via manual
   `workflow_dispatch` trigger.
2. Deployed site shows `DSM Methodology — v1.5.2` in the title.
3. Deployed site footer shows `Built from DSM v1.5.2 (2026-MM-DD)` with
   the actual build date, and a "License" link pointing at Central's
   LICENSE file.
4. `.last-built-version` file at repo root contains `v1.5.2`.
5. R3 measurement items recorded in
   `dsm-docs/research/done/2026-04-17_doc-site-best-practices_research.md`
   (update, not new file):
   - Total `mystmd build` wall time
   - Largest-page (likely DSM_3-sized) first-paint time and DOM node count
   - Search-index size
6. No scheduled cron active yet. Schedule stanza commented out in
   the workflow file with a note referencing Sprint 2.

## Work items (ordered by dependency)

### A. Initialize MyST book skeleton

- Run `myst init --gh-pages` in `book/`.
- Commit the generated `book/myst.yml`, `book/.gitignore` stub, and
  `.github/workflows/deploy.yml` (the build/deploy half of the
  pipeline generated by `myst init`).
- Add `book/content/` to `.gitignore` — this directory holds the
  copied Central files and should never be tracked.

### B. Content-copy script

- Script location: `scripts/copy-central-content.sh`.
- Behavior: takes `CENTRAL_DIR` and `DEST_DIR` arguments; copies
  `DSM_*.md`, `README.md`, `CHANGELOG.md`, `CONTRIBUTING.md`,
  `CODE_OF_CONDUCT.md` from `CENTRAL_DIR` to `DEST_DIR`. No recursion.
- Flat glob only — no `**/` patterns.

### C. Tag-filter + diff-check script

- Script location: `scripts/check-central-version.sh`.
- Behavior:
  1. Query Central's latest tag via `gh api repos/albertodiazdurana/
     dsm-agentic-ai-data-science-methodology/tags` (or equivalent
     public API call, no auth needed for public repo read).
  2. Filter through regex `^v[0-9]+\.[0-9]+\.[0-9]+$`.
  3. Read `.last-built-version` from repo root (or empty if missing).
  4. Exit 0 (no rebuild needed) if latest matches last-built.
  5. Exit 10 (rebuild needed) and print the new tag to stdout otherwise.

### D. Pre-build `myst.yml` mutator

- Script location: `scripts/inject-version.sh`.
- Behavior: takes a tag (e.g., `v1.5.2`) and a build date
  (`YYYY-MM-DD`) and rewrites `book/myst.yml`:
  - `site.options.logo_text` = `"DSM Methodology — {tag}"`
  - Footer string carrying `"Built from DSM {tag} ({date})"` plus a
    License link to Central's LICENSE file URL — mechanism TBD
    (verify-in-implementation from R3).
- If `myst.yml` alone cannot template the footer, file a follow-up BL
  for the custom-theme path rather than hand-rolling it in this
  sprint.

### E. Scheduled-build workflow

- Workflow file: `.github/workflows/scheduled-build.yml`.
- Triggers: `workflow_dispatch` (active); `schedule: cron '0 6 * * *'`
  (commented out, enabled in Sprint 2).
- Steps:
  1. Checkout this repo
  2. Clone DSM Central (public, no secret): `gh repo clone
     albertodiazdurana/dsm-agentic-ai-data-science-methodology
     /tmp/central`
  3. Run `scripts/check-central-version.sh`; if exit 0, no-op and done
  4. If rebuild needed: `git -C /tmp/central checkout {new-tag}`
  5. Run `scripts/copy-central-content.sh /tmp/central book/content/`
  6. Run `scripts/inject-version.sh {new-tag} $(date -u +%Y-%m-%d)`
  7. Run the deploy workflow generated by `myst init --gh-pages`
     (either invoke it or inline `mystmd build` + deploy-to-Pages
     action)
  8. Write `{new-tag}` to `.last-built-version` and commit + push back
     to this repo
  9. On failure: GitHub's default notification to the repo owner
     fires automatically

### F. GitHub Pages configuration

- In repo Settings → Pages, set Source = "GitHub Actions".
- Verify the `deploy.yml` from A is the pages-publishing job.
- One-time repo admin step, not a script.

### G. First end-to-end test

- Manually fire `scheduled-build.yml` via
  `gh workflow run scheduled-build.yml`.
- Expect: green run, `v1.5.2` page deployed, `.last-built-version`
  committed back.

### H. Record measurements

- Update R3 research file (now in `research/done/`) with the
  three measurement results. File was moved to `done/` in Session 1
  but update-in-done is acceptable per research lifecycle when the
  update is the outcome of a later sprint.
- If any measurement breaches the criterion from R3 (e.g., load > 3s
  on largest page), file a follow-up BL for Sprint 2.

### I. `.last-built-version` commit-back polish

- In step E.8, the commit message should be
  `chore: build DSM {tag} (YYYY-MM-DD)` so the commit is both
  meaningful and keeps the schedule alive (60-day inactivity gotcha
  mitigation).
- Commits use a bot identity or the default
  `github-actions[bot]` persona; no PAT required.

## Dependencies and prerequisites

- `mystmd` installed in the workflow runner (GitHub Actions: `uv pip
  install mystmd` or similar setup step).
- `gh` CLI available (preinstalled on GitHub-hosted runners).
- Repo Settings → Pages configured (item F).
- DSM Central stays public — any change to visibility would break
  the no-PAT assumption and require revisiting Decision 0002.

## Risks

- **`myst init --gh-pages` may generate workflow assumptions that
  conflict with the scheduled-build workflow (E).** Mitigation:
  treat the generated deploy.yml as the build/deploy step; keep our
  scheduled-build.yml as the orchestrator that triggers it. If the
  generated workflow has mandatory triggers (e.g., `on: push`), strip
  them to avoid double-builds.
- **R3 verify-in-impl items may fail criteria.** Mitigation: Sprint 1
  exit criterion 5 mandates measurement recording even when pass, so
  we know before shipping whether Sprint 2 needs corrective items.
- **First-build deploy permissions.** GitHub Pages requires specific
  `permissions:` fields in the workflow (`pages: write`, `id-token:
  write`). Mitigation: copy from the `myst init --gh-pages` template;
  don't hand-roll.
